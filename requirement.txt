===========================================================================
1. PROJECT OVERVIEW
===========================================================================

A standalone Java Spring Boot utility designed to monitor and intelligently
analyze log files generated by multiple Java Batch Jobs running on a Windows
Server environment.

When an error or exception is detected, the application automatically invokes
either Amazon Bedrock (cloud) or Ollama (local) AI engine to perform root
cause analysis and generate a human-readable remediation suggestion, which
is appended to a dedicated output file "AI_fix.txt".

Key Goals:
  - Automate log triage and reduce manual debugging time.
  - Provide AI-powered root cause analysis for Java Batch failures.
  - Support both cloud (AWS Bedrock) and on-premise (Ollama) AI providers.
  - Require zero code changes for configuration updates (fully externalized).


===========================================================================
2. SCOPE
===========================================================================

IN SCOPE:
  - Monitoring multiple Java Batch Job log directories.
  - Scanning the latest modified log file per directory.
  - Detecting errors/exceptions and extracting relevant log snippets.
  - Invoking AI (Bedrock or Ollama) with a structured prompt.
  - Writing timestamped remediation output to AI_fix.txt.
  - Packaging as a standalone executable JAR with external configuration.

OUT OF SCOPE:
  - Real-time (streaming) log monitoring (tailing).
  - Database-based logging or persistence.
  - UI or web dashboard.
  - Support for non-Windows OS (Linux/Mac) in this version.
  - Automatic execution of AI-suggested fixes.


===========================================================================
3. SYSTEM ARCHITECTURE & EXECUTION FLOW
===========================================================================

  [Windows CMD Script] (.cmd)
       |
       |  (java -jar LogAnalyzer.jar -Dspring.config.location=./config/)
       v
  [Spring Boot Application]  (Standalone JAR)
       |
       |-- Reads application.properties (external)
       |-- Scans configured log directories
       |-- Picks latest-modified log file per directory
       |-- Extracts last 50-100 lines + error context
       |
       |-- [AI Provider Switch]
       |       |-- ai.provider=bedrock  --> Amazon Bedrock (Claude-3)
       |       |-- ai.provider=ollama   --> Local Ollama API (llama3 etc.)
       |
       |-- Sends structured prompt to AI engine
       |-- Receives AI response (root cause + fix suggestion)
       v
  [AI_fix.txt]  (Appended output with timestamp)

Execution Trigger:
  - The analyzer JAR is appended to the end of each Batch Job's CMD file.
  - It runs as a post-job step automatically after the batch job completes.

Example CMD integration:
  @echo off
  call run_batch_job.cmd
  java -Dspring.config.location=C:\BatchAnalyzer\config\ -jar C:\BatchAnalyzer\log-analyzer.jar
  echo Batch Job + AI Log Analysis Complete.


===========================================================================
4. FUNCTIONAL REQUIREMENTS
===========================================================================

---------------------------------------------------------------------------
FR-1: External Configuration (MANDATORY)
---------------------------------------------------------------------------
  - All configurable parameters must be externalized in application.properties.
  - The JAR must load configuration from an external directory using:
      -Dspring.config.location=<path-to-config-folder>
  - This allows changes to log paths, AI model, and output directories
    without rebuilding the JAR.
  - Supported config keys must include: AI provider, model names, log
    directory paths, output file path, and scan depth (lines to read).

---------------------------------------------------------------------------
FR-2: Multi-Application Log Directory Monitoring
---------------------------------------------------------------------------
  - The application must support scanning one or more log directories,
    each mapped to a logical application name in the config file.
  - Config Format (list-style):
      log.apps[0].name=CoreBankingJob
      log.apps[0].path=C:/Batch/Logs/Banking/
      log.apps[1].name=InventorySync
      log.apps[1].path=D:/Batch/Logs/Inventory/
  - Logic: For each configured directory, identify and read only the
    most recently modified log file.
  - If a directory does not exist or is empty, log a warning and skip it.

---------------------------------------------------------------------------
FR-3: Hybrid AI Engine (Switchable at Runtime)
---------------------------------------------------------------------------
  - A single property (ai.provider.choice) determines which AI engine to use.
  - Supported values: "ollama" | "bedrock"
  - Implementation must use an abstraction layer:
      Interface: AIService
          + String analyzeLog(String logContent, String appName)
      Implementations:
          - OllamaAIServiceImpl  (uses local HTTP REST API)
          - BedrockAIServiceImpl (uses AWS SDK for Java v2)
  - Spring Bean selection based on ai.provider.choice property (using
    @ConditionalOnProperty or @Profile).

  Ollama Integration:
  - Base URL: http://localhost:11434 (configurable)
  - API Endpoint: POST /api/generate
  - Request Body: { "model": "<model>", "prompt": "<prompt>", "stream": false }
  - Timeout: 30 seconds (configurable)

  Amazon Bedrock Integration:
  - Uses AWS SDK for Java v2 (BedrockRuntimeClient)
  - Model: anthropic.claude-3-sonnet or configurable model ID
  - Authentication: Via AWS environment variables (AWS_ACCESS_KEY_ID,
    AWS_SECRET_ACCESS_KEY, AWS_DEFAULT_REGION)
  - Request format: Anthropic Messages API (Claude native format)

---------------------------------------------------------------------------
FR-4: Error Extraction & AI Prompting
---------------------------------------------------------------------------
  - Read the last N lines (default: 100, configurable) from the latest
    log file per configured application.
  - Scan the extracted lines for keywords: "ERROR", "EXCEPTION",
    "FATAL", "Caused by", "Stack trace", "WARN" (severity configurable).
  - If no error keywords found: skip AI call, log info message.
  - If errors detected: build a structured prompt and invoke the AI engine.

  Prompt Template:
  -----------------------------------------------------------------------
  You are a Java backend expert. Analyze the following Spring Batch job
  error log from a Windows Server environment.

  Application: {appName}
  Log Excerpt:
  ---
  {logContent}
  ---
  Tasks:
  1. Identify the root cause of the error.
  2. Suggest a specific, actionable fix for a Windows Server environment.
  3. Mention any configuration or environment changes if required.
  -----------------------------------------------------------------------

---------------------------------------------------------------------------
FR-5: Remediation Output File (AI_fix.txt)
---------------------------------------------------------------------------
  - AI response must be appended (not overwritten) to the output file.
  - Output file path must be configurable via: output.fix.file=<path>
  - Output Format per entry:

  ========================================================================
  [YYYY-MM-DD HH:mm:ss] | Application: CoreBankingJob | Error: NullPointerException
  ------------------------------------------------------------------------
  Root Cause Analysis:
  <AI Response - Root Cause Section>

  Suggested Fix:
  <AI Response - Fix Section>

  Environment Notes:
  <AI Response - Environment/Config Notes>
  ========================================================================

  - If the AI call fails, write a failure notice with the error message
    instead of the AI response, so the file always has an audit trail.

---------------------------------------------------------------------------
FR-6: Execution Summary Logging (Console)
---------------------------------------------------------------------------
  - At the end of execution, print a summary to the console (and application
    log) showing:
      * Total directories scanned
      * Number of log files read
      * Number of errors detected
      * Number of successful AI calls
      * Number of failed AI calls
      * Output file path
  - Example:
      ---- AI Log Analyzer Summary ----
      Directories Scanned : 3
      Errors Detected     : 2
      AI Calls Successful : 2
      AI Calls Failed     : 0
      Output File         : C:/Batch/AI_Reports/AI_fix.txt
      ---------------------------------

6. CONFIGURATION SCHEMA (application.properties)
===========================================================================

  # -----------------------------------------------------------------------
  # AI Provider Selection
  # -----------------------------------------------------------------------
  ai.provider.choice=ollama              # Options: ollama | bedrock

  # -----------------------------------------------------------------------
  # Ollama Settings (used when ai.provider.choice=ollama)
  # -----------------------------------------------------------------------
  ollama.base-url=http://localhost:11434
  ollama.model=llama3
  ollama.timeout-seconds=30

  # -----------------------------------------------------------------------
  # Amazon Bedrock Settings (used when ai.provider.choice=bedrock)
  # -----------------------------------------------------------------------
  aws.bedrock.model-id=anthropic.claude-3-sonnet-20240229-v1:0
  aws.region=us-east-1

  # -----------------------------------------------------------------------
  # Log Scanning Configuration
  # -----------------------------------------------------------------------
  log.scan.lines=100                     # Number of lines to read from end
  log.scan.error-keywords=ERROR,EXCEPTION,FATAL,Caused by

  # Multiple Application Log Directory Mappings
  log.apps[0].name=CoreBankingJob
  log.apps[0].path=C:/Batch/Logs/Banking/

  log.apps[1].name=InventorySync
  log.apps[1].path=D:/Batch/Logs/Inventory/

  log.apps[2].name=ReportGenerator
  log.apps[2].path=E:/Batch/Logs/Reports/

  # -----------------------------------------------------------------------
  # Output File Configuration
  # -----------------------------------------------------------------------
  output.fix.file=C:/Batch/AI_Reports/AI_fix.txt

7. DEPLOYMENT & PACKAGING
===========================================================================

  Packaging:
  - Build as a standalone executable JAR using Maven:
      mvn clean package -DskipTests
  - JAR output: target/log-analyzer-1.0.0.jar

  Directory Structure on Windows Server:
  C:/BatchAnalyzer/
    |-- log-analyzer-1.0.0.jar
    |-- config/
    |     |-- application.properties
    |-- run_analyzer.cmd              (wrapper script)
    |-- logs/
    |     |-- analyzer.log            (internal application log)

  Wrapper CMD Script (run_analyzer.cmd):
  -----------------------------------------------------------------------
  @echo off
  SET JAR_PATH=C:\BatchAnalyzer\log-analyzer-1.0.0.jar
  SET CONFIG_PATH=C:\BatchAnalyzer\config\
  SET LOG_FILE=C:\BatchAnalyzer\logs\analyzer.log

  echo Starting AI Log Analyzer...
  java -Dspring.config.location=%CONFIG_PATH% ^
       -Dlogging.file.name=%LOG_FILE% ^
       -jar %JAR_PATH%

  echo AI Log Analyzer execution complete. Exit Code: %ERRORLEVEL%
  -----------------------------------------------------------------------

8. NON-FUNCTIONAL REQUIREMENTS
===========================================================================

  NFR-1: Security
  - AWS credentials must NEVER be hardcoded in source code or properties.
  - Use Windows Environment Variables for AWS authentication:
      AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_DEFAULT_REGION
  - application.properties must not contain any secrets.

  NFR-2: Performance
  - Total execution time for Ollama (local) analysis: <= 30 seconds.
  - Total execution time for Bedrock (cloud) analysis: <= 60 seconds.
  - Each log file read operation should not exceed 5 seconds.

  NFR-3: Reliability & Error Handling
  - If the AI engine is unreachable or returns an error:
      * Log the error with full stack trace (at ERROR level).
      * Write a failure notice to AI_fix.txt (do not silently discard).
      * Continue processing remaining log directories (fail-safe).
  - If a log directory does not exist: log a WARNING and skip.
  - If a log directory is empty: log an INFO and skip.
  - Application must always exit with a clear status code:
      0 = Success (no errors found OR AI fix generated)
      1 = Partial failure (some AI calls failed)
      2 = Fatal error (configuration error or all AI calls failed)

  NFR-4: Maintainability
  - All AI provider logic must be behind the AIService interface.
  - Adding a new AI provider requires only a new implementation class
    and a configuration entry — no changes to core scanning logic.
  - Code must follow standard Java naming conventions and include Javadoc
    for all public classes and methods.

  NFR-5: Logging Standards
  - Use SLF4J with Logback for all internal logging.
  - Log levels:
      INFO  : Normal flow (scanning started, file selected, AI called)
      WARN  : Non-critical issues (empty directory, no error found)
      ERROR : AI call failure, file read failure, config issues
  - Log format: [YYYY-MM-DD HH:mm:ss] [LEVEL] [Class] - Message

  NFR-6: Portability
  - While the primary target is Windows Server, the code must avoid
    OS-specific hardcoded paths (use configurable properties only).
  - File separators must use Java's File.separator or Path API.
